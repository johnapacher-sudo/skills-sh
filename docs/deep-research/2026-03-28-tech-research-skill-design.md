# 技术深度调研 Skill 设计文档

**日期：** 2026-03-28
**状态：** 已批准

## 概述

创建一套多 Skill 流水线架构的技术深度调研工具，用于自动化调研新技术（以 AI 相关新技术/新产品/生产力工具为主）。用户只需提供技术名称或关键词，即可获得包含完整调研报告和硬核学习教程的 Markdown 文档。

## 架构设计

### 4+1 Skill 流水线

```
tech-research（总调度）
    ├── tech-discover（技术发现与检索）
    ├── tech-analyze（深度分析）
    ├── tech-tutorial（硬核教程生成）
    └── tech-review（内容质量审阅）
```

### 各 Skill 职责

#### 1. tech-discover（技术发现与检索）

**触发词：** "快速了解一下 xxx"、"xxx 是什么"、"发现 xxx 的信息"

**职责：**
- WebSearch 搜索该技术的官方文档、GitHub 仓库、社区评价
- WebFetch 抓取关键页面内容（官网、README、Getting Started）
- 整理出技术的基础信息清单
- 输出结构化的「技术概览」中间文件

**执行流程：**
1. 根据用户提供的技术名称/关键词/GitHub 链接，进行多维度搜索
2. 从至少 3 个独立来源获取信息（官网、GitHub、技术博客/社区）
3. 抓取关键页面详细内容
4. 整理并结构化输出

**输出：** `research/<tech-name>/01-discovery.md`

#### 2. tech-analyze（深度分析）

**触发词：** "深度分析 xxx"、"对比 xxx 和 yyy"、"分析 xxx 的原理"

**职责：**
- 基于概览，深入分析技术原理、架构设计、核心概念
- 对比同类技术（横向对比分析）
- 分析适用场景、优缺点、生态成熟度
- 输出完整调研报告

**执行流程：**
1. 读取 discovery 报告，识别需要深入了解的关键点
2. 搜索补充信息（源码分析、架构文档、技术博客深度文章）
3. 分析技术原理和架构设计
4. 进行同类技术横向对比
5. 评估适用场景和生产就绪度
6. 输出完整分析报告

#### 3. tech-tutorial（硬核教程生成）

**触发词：** "出一份 xxx 的教程"、"生成 xxx 学习教程"、"xxx 教程"

**职责：**
- 基于调研报告，生成从零到一的完整学习教程
- 覆盖所有核心技术点，每个点配有完整代码示例
- 包含实战练习和最佳实践
- 按难度递进组织（入门→进阶→高级）

**执行流程：**
1. 读取 discovery 和 analysis 报告
2. 梳理所有核心技术知识点
3. 设计教程结构（入门→进阶→高级→实战）
4. 为每个知识点编写讲解和完整代码示例
5. 设计贯穿教程的实战项目
6. 编写常见问题和排查指南

#### 4. tech-review（内容质量审阅）

**触发词：** "审阅 xxx 的调研报告"、"检查调研质量"、"review xxx 调研"

**职责：**
- 审阅所有已生成的文档，检查准确性、完整性、一致性
- 验证代码示例是否正确可运行
- 检查教程的逻辑递进是否合理
- 发现信息缺失或不够深入的地方，补充完善
- 确保技术术语使用准确、前后一致

**执行流程：**
1. 读取所有已生成的文档（01-03）
2. 按审阅清单逐项检查
3. 对发现的问题分级（P0/P1/P2）
4. 修复所有 P0（事实错误）和 P1（重要遗漏）
5. 输出审阅报告和修正说明

**输出：** `research/<tech-name>/04-review.md`

**审阅清单：**
- [ ] 事实准确性：关键声明是否有来源支撑
- [ ] 代码可运行性：示例代码是否可直接执行
- [ ] 完整性：是否覆盖了该技术的所有核心概念
- [ ] 逻辑递进：教程章节顺序是否合理
- [ ] 术语一致性：全文术语使用是否统一
- [ ] 时效性：信息是否为最新版本
- [ ] 来源可溯：关键信息是否标注了来源

**问题分级：**
- P0（必须修复）：事实错误、代码无法运行、严重遗漏
- P1（应当修复）：重要信息不完整、代码示例不够清晰、对比分析缺失维度
- P2（建议改进）：表述优化、排版改进、补充推荐资源

#### 5. tech-research（总调度入口）

**触发词：** "调研 xxx"、"全面调研 xxx"、"深度研究 xxx"、"research xxx"

**职责：**
- 作为总调度，按顺序执行 discover → analyze → tutorial → review
- 管理调研目录和文件命名
- 汇总最终输出

**执行流程：**
1. 创建 `research/<tech-name>/` 目录
2. 执行 tech-discover，生成 01-discovery.md
3. 执行 tech-analyze，生成 02-analysis.md
4. 执行 tech-tutorial，生成 03-tutorial.md
5. 执行 tech-review，生成 04-review.md
6. 如 review 发现 P0/P1 问题，回溯修复相关文档
7. 输出完成汇总

## 质量保证机制

### discover 阶段
- **多源交叉验证**：至少从 3 个独立来源获取信息
- **信息溯源**：每个关键信息标注来源链接
- **时效性检查**：标注信息获取日期，标记过时信息
- **禁止编造**：不确定的信息标注为"待验证"，禁止猜测

### analyze 阶段
- **代码示例必须可运行**：基于官方文档或实际测试
- **版本对齐**：标注分析所基于的具体版本号
- **对比分析用表格**：确保对比维度一致、数据可查
- **不确定性标注**：对不确信的分析结论标注置信度

### tutorial 阶段
- **循序渐进**：每个知识点必须建立在前一个之上
- **代码示例完整**：每个示例都是可直接运行的完整代码，不写伪代码
- **实战导向**：每章结束有练习题，整个教程有贯穿的实战项目
- **避坑指南**：标注常见错误和调试方法

### review 阶段
- 执行完整审阅清单
- 所有 P0/P1 问题必须修复后才能输出最终版
- 输出质量评分

- **语言：** 全中文

- **存放位置：** `research/<tech-name>/` 目录
- **文件命名：** 阶段编号 + 阶段名称

## 技术类型覆盖

- 框架/语言（React、Vue、Rust 等）
- 协议/工具链（gRPC、OpenTelemetry 等）

- AI 相关新产品和生产力工具

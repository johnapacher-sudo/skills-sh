# PRD 需求处理流水线 Skill 设计文档

**日期：** 2026-04-03
**状态：** 已批准

## 概述

创建一套 3 个 Skill 的 PRD 需求处理流水线，将复杂的 PRD 文档自动去噪、拆分、澄清，最终产出清晰的需求清单。用户只需提供 PRD（文本或文件），即可获得按前端页面和后端模块拆分、经过多轮澄清确认的完整需求清单。

## 架构设计

### 3 Skill 流水线

```
prd-research（总调度）
    ├── prd-parse（PRD 去噪 + 需求拆分）
    ├── prd-clarify（需求澄清）
    └── 最终汇总输出
```

### 数据流

```
用户输入 PRD（文本/文件）
       ↓
  [prd-research 总调度]
       ↓
  [prd-parse]
    - 读取 PRD 内容
    - 过滤干扰信息
    - 按前端页面/后端模块拆分
    - 输出独立需求文件 + 索引文件
       ↓
  用户确认拆分是否合理
       ↓  (不合理 → prd-parse 重新拆分)
  [prd-clarify]
    - 提取所有子需求的不确定项
    - 生成批量待澄清清单
    - 用户批量回复
    - 更新各子需求文件
    - 循环直到无待澄清项
       ↓
  [汇总输出]
    - 合并所有已澄清的需求
    - 输出最终需求清单
```

### 输出目录结构

```
requirements/
└── <project-name>/
    ├── 00-prd-raw.md                  # 去噪后的原始 PRD + 过滤附录
    ├── 01-split-overview.md           # 拆分总览索引文件
    ├── fe-<page-name>.md              # 前端需求文件（按页面）
    ├── be-<module-name>.md            # 后端需求文件（按模块）
    ├── cm-<name>.md                   # 通用需求文件
    ├── 02-clarify-round-1.md          # 第一轮澄清记录
    ├── 02-clarify-round-2.md          # 第二轮澄清记录（如需）
    └── 03-final.md                    # 最终需求清单
```

**文件命名规则：**
- 前端：`fe-<page-name>.md`（如 `fe-user-login.md`、`fe-order-list.md`）
- 后端：`be-<module-name>.md`（如 `be-auth-service.md`、`be-order-api.md`）
- 通用：`cm-<name>.md`（如 `cm-permission.md`）
- 项目名从 PRD 内容中提取或由用户指定

## 各 Skill 详细设计

### 1. prd-parse（PRD 去噪 + 需求拆分）

**触发词：** "解析这个 PRD"、"拆分需求"、"分析需求文档"

**输入：**
- 用户直接粘贴的 PRD 文本
- 或用户提供文件路径（支持 .md / .txt），通过 Read 工具读取

**执行流程：**

**第一步：读取输入**
- 检测输入类型（文本 / 文件路径）
- 文件路径：用 Read 工具读取内容
- 直接文本：直接处理

**第二步：去噪过滤**
- 识别并过滤以下干扰信息：
  - 市场背景、行业分析
  - 商业策略、竞品分析
  - 项目进度计划、里程碑
  - 非功能性约束中的泛泛描述（如"系统要稳定"）
  - 冗余的管理性描述
- 保留以下核心信息：
  - 功能描述
  - 交互说明、UI 描述
  - 接口需求
  - 数据需求、数据模型
  - 页面/模块描述
  - 业务规则、计算逻辑
  - 权限、角色、状态流转
- 过滤内容不丢弃，单独存入 `00-prd-raw.md` 的附录区

**第三步：需求拆分**
- **前端需求**：按页面维度拆分
  - 每个页面一个独立文件
  - 识别页面间的关联和跳转关系
  - 包含：页面名称、功能描述、交互要点、UI 元素、依赖的后端接口
- **后端需求**：按功能模块拆分
  - 每个模块一个独立文件
  - 包含：模块名称、功能描述、接口设计要点、数据模型、业务规则
- **通用需求**：跨前后端的需求单独列出
  - 包含：权限控制、国际化、主题配置等

**第四步：生成输出**
- 创建 `requirements/<project-name>/` 目录
- 生成 `00-prd-raw.md`（去噪后 PRD + 过滤附录）
- 为每个子需求生成独立文件（使用统一需求模板）
- 生成 `01-split-overview.md` 索引文件

**输出模板 - 索引文件（01-split-overview.md）：**

```markdown
# <项目名称> - 需求拆分总览

## 前端需求
- [FE-001 <页面名称>](fe-<page-name>.md)
- [FE-002 <页面名称>](fe-<page-name>.md)

## 后端需求
- [BE-001 <模块名称>](be-<module-name>.md)

## 通用需求
- [CM-001 <需求名称>](cm-<name>.md)

## 待用户确认
- [ ] 拆分粒度是否合适？
- [ ] 是否有遗漏的页面或模块？
- [ ] 前后端边界是否准确？
```

**输出模板 - 子需求文件：**

```markdown
# <编号>: <需求名称>

## 基本信息
- **类型：** 前端/后端/通用
- **来源：** PRD 第 X 节 / 第 X-Y 页
- **优先级：** 待定

## 需求描述
<清晰描述该需求要做什么>

## 功能清单
1. <功能1>
2. <功能2>

## 验收标准
- [ ] <标准1>
- [ ] <标准2>

## 不确定项
- [ ] <待澄清问题1>
- [ ] <待澄清问题2>
```

**编号系统：**
- FE-NNN：前端需求（按顺序编号）
- BE-NNN：后端需求
- CM-NNN：通用需求

**质量约束：**
- 每个子需求必须标注来源（PRD 的哪一节）
- 不确定的内容标记为"不确定项"，禁止猜测
- 过滤内容保留在附录区，不丢弃
- 去噪保留率应 > 90% 的需求相关内容

---

### 2. prd-clarify（需求澄清）

**触发词：** "澄清需求"、"整理待澄清项"、"确认需求"

**输入：**
- 读取 `requirements/<project-name>/01-split-overview.md` 获取文件列表
- 逐个读取每个子需求文件

**执行流程：**

**第一步：收集不确定项**
- 遍历所有子需求文件的"不确定项"部分
- 检查"需求描述"和"功能清单"中模糊的描述，补充为不确定项
- 模糊描述识别标准：
  - 缺少具体数值（如"大量用户"）
  - 缺少明确条件（如"适当时候"）
  - 缺少边界定义（如"相关数据"）
  - 互相矛盾的描述

**第二步：生成待澄清清单**

```markdown
# 需求澄清 - 第 N 轮

## 待澄清项汇总

### FE-001: <页面名称>
- [ ] **Q1:** <具体问题>
- [ ] **Q2:** <具体问题>

### BE-001: <模块名称>
- [ ] **Q3:** <具体问题>

---

## 请逐项回复
> 在对应问题后直接填写回答，不需要的可以标注"不需要"
```

保存到 `02-clarify-round-N.md`

**第三步：等待用户回复**

**第四步：处理回复**
- 已回答的问题 → 从子需求文件的"不确定项"中移除
- 回答内容补充到子需求文件的对应位置（需求描述/功能清单/验收标准）
- 回答中发现的新不确定项 → 添加到子需求文件
- 用户标注"不需要" → 从需求中移除，记录到澄清文件
- 更新索引文件状态

**第五步：判断是否继续**
- 所有子需求文件的"不确定项"都为空 → 结束，通知 prd-research 汇总
- 仍有不确定项 → 生成 `02-clarify-round-N+1.md`，回到第三步

**质量约束：**
- 每个澄清问题必须具体、可回答（禁止模糊问题）
- 澄清回答必须直接更新到对应子需求文件
- 保留所有澄清轮次记录
- 每轮澄清的问题数不超过 15 个（超出则拆分到下一轮）

---

### 3. prd-research（总调度）

**触发词：** "处理这个 PRD"、"整理需求"、"分析 PRD"、"需求分析"

**执行流程：**

**Phase 1 - 解析拆分：**
1. 从用户输入中提取 PRD 内容（文本或文件路径）
2. 提取或询问项目名称
3. 调用 `prd-parse` 执行去噪 + 拆分
4. 展示拆分总览（`01-split-overview.md` 内容）
5. 询问用户是否确认拆分
   - 确认 → 进入 Phase 2
   - 不确认 → 根据反馈调整拆分，重新展示

**Phase 2 - 多轮澄清：**
6. 调用 `prd-clarify` 生成本轮待澄清清单
7. 展示清单，等待用户批量回复
8. 更新各子需求文件
9. 如仍有不确定项，回到步骤 6
10. 澄清完成时通知用户

**Phase 3 - 汇总输出：**
11. 读取所有子需求文件
12. 生成最终需求清单 `03-final.md`

**最终需求清单模板（03-final.md）：**

```markdown
# <项目名称> - 最终需求清单

## 概览
- **前端页面数：** N
- **后端模块数：** M
- **通用需求数：** K
- **总需求数：** T
- **澄清轮次：** R

---

## 前端需求
- [FE-001 <页面名称>](fe-<page-name>.md) - 已澄清
- [FE-002 <页面名称>](fe-<page-name>.md) - 已澄清

## 后端需求
- [BE-001 <模块名称>](be-<module-name>.md) - 已澄清

## 通用需求
- [CM-001 <需求名称>](cm-<name>.md) - 已澄清

---

## 需求统计
| 类型 | 数量 | 已澄清 | 待澄清 |
|------|------|--------|--------|
| 前端 | N | N | 0 |
| 后端 | M | M | 0 |
| 通用 | K | K | 0 |
| **合计** | **T** | **T** | **0** |

---

## 澄清历史
- [第 1 轮澄清](02-clarify-round-1.md) - 解决 X 个问题
- [第 2 轮澄清](02-clarify-round-2.md) - 解决 Y 个问题
```

**质量约束（贯穿全流程）：**
- 全中文输出
- 禁止编造需求，PRD 中没有的内容标注为"PRD 未提及"
- 每个需求可追溯来源
- 澄清问题具体可回答
- 所有子需求文件使用统一模板

```

└── skills/
    ├── prd-parse/
    │   ├── SKILL.md
    │   └── references/

    │       └── split-overview-template.md # 索引文件模板
    ├── prd-clarify/
    │   ├── SKILL.md
    │   └── references/
    │       └── clarify-template.md       # 澄清清单模板
    └── prd-research/
        ├── SKILL.md
        └── references/
            └── final-template.md         # 最终需求清单模板
```

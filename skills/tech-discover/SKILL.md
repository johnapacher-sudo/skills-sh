# 技术发现（tech-discover）

## 概述

本 Skill 用于对一项技术进行多源检索和信息收集，产出一份数据准确、来源清晰、结构化的技术发现报告。当用户提到"快速了解 xxx"、"xxx 是什么"、"发现 xxx 的信息"、"介绍一下 xxx"、"xxx 概览"等意图时，自动触发本技能执行流程。

核心目标：在 10-15 分钟的交互窗口内，通过多维度搜索、关键页面抓取和交叉验证，帮助用户建立对一项新技术的完整认知框架。

## 执行流程

按以下步骤严格执行，每一步都必须完成后才能进入下一步。

### 第一步：解析用户输入

从用户的消息中提取以下信息：

- **技术名称**：确定用户要了解的技术、框架、库、工具或平台的准确名称。注意区分同名但不同的项目（例如 Apollo GraphQL 和 Apollo Router）。
- **关键词**：提取用户提到的附加限定词，如语言偏好、版本要求、使用场景等。
- **URL（如有）**：如果用户直接提供了一个链接，记录该链接作为优先信息来源。

如果用户提供的名称存在歧义，先向用户确认具体指代的技术，再继续后续步骤。

### 第二步：执行多维度搜索

使用 WebSearch 工具，构造至少以下 3 个维度的搜索查询：

1. **官方信息搜索**：`{{技术名称}} official site` 或 `{{技术名称}} 官网`。目标是找到官方网站、文档站点和项目主页。
2. **代码仓库搜索**：`{{技术名称}} github`。目标是找到 GitHub 仓库地址，获取 Stars、贡献者、最近提交等数据。
3. **社区评价搜索**：`{{技术名称}} tutorial review comparison 2025` 或 `{{技术名称}} 教程 评测 对比`。目标是找到技术博客、评测文章、对比分析等第三方信息来源。

根据搜索结果的质量，可追加以下补充搜索维度：

- `{{技术名称}} vs {{竞品}} comparison` —— 竞品对比，帮助定位技术栈位置
- `{{技术名称}} changelog release notes` —— 版本历史和更新节奏
- `{{技术名称}} reddit hacker news` —— 社区真实反馈和讨论

每个搜索查询执行后，记录返回的所有相关链接和摘要信息，用于后续步骤。

### 第三步：抓取关键页面内容

基于第二步搜索到的链接，使用 mcp__web_reader__webReader 工具抓取以下关键页面的完整内容：

1. **官网首页**：获取技术定位描述、核心特性列表、版本号、下载/安装方式等信息。
2. **GitHub README**：获取项目描述、Stars 数、License、构建状态、快速入门指南等结构化信息。如果 GitHub 页面内容过长，优先读取 README 的前 500 行。
3. **文档 Getting Started 页面**：获取安装命令、最小可运行示例、核心概念说明。这有助于理解技术的使用方式和学习曲线。

抓取时的注意事项：

- 如果某个页面无法访问或返回错误，记录该情况并在最终报告中标注"该页面无法访问"。
- 优先抓取官方来源的内容，第三方来源作为补充和交叉验证。
- 每个页面抓取后，立即提取其中的关键数据点（名称、版本、License、Stars 等），不要等到最后再整理。

### 第四步：交叉验证信息

将第三步从多个来源提取的信息进行对比和交叉验证：

- **核心数据点验证**：项目名称、当前版本号、License 类型、主要编程语言等基础信息，必须至少从 2 个独立来源确认一致。
- **不一致处理**：如果不同来源提供的信息存在矛盾（例如版本号不一致），在报告中标注"待验证"，并列出所有找到的不同值及其来源。
- **时效性检查**：关注信息的发布日期。超过 1 年的信息需要标注"信息可能过时"，并尝试寻找更新的来源。
- **项目活跃度判断**：通过 GitHub 最近提交日期、Issue 回复速度、Release 频率等指标判断项目是否仍在积极维护。如果项目已停止维护或归档，必须在报告开头醒目标注。

### 第五步：按模板格式化输出

读取 `references/output-template.md` 模板文件，按照模板中的所有章节填写内容。填写规则：

- 每一个章节都必须填写，不允许留空。如果某个章节确实无法获取到信息，填写"未找到相关信息"并说明原因。
- 模板中以 `{{}}` 包裹的占位符替换为实际获取到的数据。
- 核心特性部分至少列出 3 条，每条包含特性名称和简要说明。
- 关键链接汇总部分，每个链接必须填写完整的 URL 和描述性的链接文本。
- 信息来源部分，列出所有实际使用的来源链接，每个链接标注信息获取日期（使用当天日期）。

### 第六步：保存输出文件

将格式化后的报告保存到以下路径：

```
research/{{tech-name}}/01-discovery.md
```

其中 `{{tech-name}}` 使用技术名称的小写英文形式，单词之间用连字符连接。例如：

- React → `research/react/01-discovery.md`
- Kubernetes → `research/kubernetes/01-discovery.md`
- Node.js → `research/nodejs/01-discovery.md`

保存前执行以下操作：

- 使用 Bash 工具创建 `research/{{tech-name}}/` 目录（如不存在）。
- 确认文件内容完整，所有模板章节均已填写。
- 保存完成后，向用户输出文件路径，并简要概括报告中的关键发现。

## 质量约束

以下规则为硬性要求，任何情况下不得违反：

### 信息来源要求

- 至少从 3 个独立来源获取信息。独立来源的定义：不同的域名和不同的内容发布方。例如官网（example.com）和 GitHub（github.com）算 2 个独立来源；同一博客的两篇文章只算 1 个来源。
- 每个关键信息（版本号、License、Stars 数、发布日期等）必须标注信息来源链接。
- 所有来源链接必须是实际访问过的有效链接，禁止凭记忆或推测填写链接。

### 数据准确性要求

- 禁止编造任何信息。如果搜索和抓取未能获取到某项数据，明确标注"待验证"，而不是猜测或填写不确定的值。
- 数值型数据（Stars 数、下载量等）标注获取时的具体数值和获取日期，因为这些数据会随时间变化。
- 版本号必须是最新的稳定版本。如果搜索结果中同时出现多个版本号，以官方来源和 GitHub Release 页面为准。

### 时效性要求

- 标注信息获取日期为执行搜索的当天日期。
- 如发现项目已停止维护、归档或不再更新，在报告开头用醒目方式标注：
  ```
  > ⚠️ 注意：该项目已于 {{日期}} 停止维护/归档，不建议用于新项目。
  ```
- 如果最近一次更新超过 6 个月，在社区生态章节标注"项目更新频率较低，需评估活跃度"。

### 输出完整性要求

- 模板中的所有章节都必须包含在最终输出中。
- 每个章节都有实质内容，不允许仅包含占位符。
- 信息来源章节至少列出 3 条实际使用的来源。

### 术语一致性要求

- 同一技术概念在报告中使用统一的名称，避免中英文混用或使用多种译名。
- 首次出现非中文技术术语时，附上英文原文。
- 技术名称以官方名称为准，不自行创造缩写或简称。

## 输出规范

### 输出文件

- 路径：`research/{{tech-name}}/01-discovery.md`
- 格式：Markdown，UTF-8 编码
- 模板：严格遵循 `references/output-template.md` 的章节结构

### 向用户展示

保存文件后，向用户输出以下内容：

1. 报告文件的保存路径。
2. 一段 3-5 句话的关键发现摘要，涵盖：该技术是什么、核心优势、社区活跃度、是否推荐使用。
3. 如果有信息标注为"待验证"，列出这些条目，建议用户进一步确认。

### 后续建议

如果用户需要更深入的信息，可建议以下方向：

- 技术深度调研（使用 tech-research skill）
- 与竞品的详细对比分析
- 实际使用体验和最佳实践
- 技术选型评估

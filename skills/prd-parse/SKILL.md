# PRD 解析与需求拆分（prd-parse）

## 概述

本 Skill 用于对 PRD（Product Requirements Document，产品需求文档）进行去噪过滤和需求拆分，按前端页面维度和后端模块维度将需求拆分为独立文件，产出结构化的需求拆分结果。当用户提到"解析这个 PRD"、"拆分需求"、"分析需求文档"、"整理 PRD"等意图时，自动触发本技能执行流程。

核心目标：从冗长杂乱的 PRD 文档中提取核心需求信息，过滤干扰内容，将需求按前端页面和后端模块进行结构化拆分，生成独立的、可追踪的需求文件，并为每个需求标注来源、验收标准和不确定项，便于后续开发排期和技术方案设计。

## 执行流程

按以下步骤严格执行，每一步都必须完成后才能进入下一步。

### 第一步：读取输入

检测用户提供的输入类型，区分以下两种情况：

- **文件路径输入**：如果用户提供的输入以 `.md` 或 `.txt` 结尾，且路径格式符合文件路径特征，使用 Read 工具读取文件内容。读取前先用 Glob 工具确认文件是否存在。
- **直接粘贴文本**：如果用户直接在对话中粘贴了 PRD 内容，直接作为待处理文本使用。
- **无有效输入**：如果输入既非文件路径也非 PRD 内容（例如仅有一句话描述），提示用户提供完整的 PRD 文档或文件路径，并说明本技能需要的输入格式。

读取完成后，对 PRD 内容进行初步浏览，识别文档的整体结构和章节划分。

### 第二步：去噪过滤

对原始 PRD 内容进行去噪处理，区分核心信息与干扰信息。

**过滤以下干扰信息：**

- 市场背景描述（宏观趋势、行业规模等）
- 行业分析内容（市场格局、发展预测等）
- 商业策略描述（商业模式、盈利计划等）
- 竞品分析内容（竞品功能对比、市场占有率等）
- 项目进度计划（里程碑、排期表、甘特图等）
- 非功能性约束中的泛泛描述（如"系统需要稳定"、"要保证安全性"等无具体标准的描述）
- 冗余的管理性描述（审批流程、文档版本记录等）

**保留以下核心信息：**

- 功能描述与业务规则
- 交互说明与操作流程
- UI 描述（页面布局、组件、样式要求）
- 接口需求与数据需求
- 数据模型与字段定义
- 页面/模块描述及其关联关系
- 计算逻辑与公式
- 权限控制与角色定义
- 状态流转与业务流程
- 具体的非功能性指标（如"页面加载时间 < 2 秒"）

**关键要求：** 过滤掉的内容不丢弃，全部保留在 `00-prd-raw.md` 文件的附录区，标注"以下为过滤内容"，确保所有原始信息可追溯。去噪保留率应大于 90% 的需求相关内容。

### 第三步：需求拆分

对去噪后的 PRD 内容进行结构化拆分，按以下维度组织：

**前端需求（按页面维度拆分）：**

- 每个页面生成一个独立的 `fe-<page-name>.md` 文件
- 文件命名使用 kebab-case 格式，例如 `fe-user-login.md`、`fe-order-list.md`
- 每个前端需求文件包含：页面名称、功能描述、交互要点、UI 元素说明、依赖的后端接口
- 识别页面之间的关联关系和跳转逻辑，在需求描述中明确标注

**后端需求（按功能模块拆分）：**

- 每个功能模块生成一个独立的 `be-<module-name>.md` 文件
- 文件命名使用 kebab-case 格式，例如 `be-user-auth.md`、`be-order-service.md`
- 每个后端需求文件包含：模块名称、功能描述、接口设计要点（请求方法、路径、参数、响应）、数据模型定义、业务规则说明

**通用需求（跨前后端的公共需求）：**

- 单独列出，生成 `cm-<name>.md` 文件
- 典型内容包括：权限控制方案、国际化（i18n）要求、日志规范、通用错误处理等
- 每个通用需求文件包含：需求名称、适用范围、具体规则说明

**编号系统：**

- 前端需求编号：FE-NNN（如 FE-001、FE-002）
- 后端需求编号：BE-NNN（如 BE-001、BE-002）
- 通用需求编号：CM-NNN（如 CM-001、CM-002）
- 编号按拆分顺序依次递增，每个分类独立编号

**拆分原则：**

- 每个子需求应具有独立性和完整性，可以独立理解和独立验收
- 如果一个页面功能过于复杂，可按功能区块进一步拆分为多个子需求
- 跨模块的公共逻辑提取为通用需求，避免重复描述

### 第四步：生成输出

按以下规范创建输出文件：

1. **创建目录结构**：使用 Bash 工具创建 `requirements/<project-name>/` 目录，其中 `<project-name>` 从 PRD 内容中提取项目名称，转换为 kebab-case 格式。

2. **生成原始文档**：创建 `00-prd-raw.md` 文件，包含去噪后的 PRD 核心内容，以及附录区的过滤内容。附录区使用分隔线明确分隔，标注每个过滤段落对应的原始章节。

3. **生成子需求文件**：读取 `references/requirement-template.md` 模板文件，为每个拆分出的子需求生成独立文件。使用 Glob 工具确认模板文件存在。模板中的 `{{占位符}}` 替换为实际内容，不确定的内容填入"不确定项"章节。

4. **生成索引文件**：读取 `references/split-overview-template.md` 模板文件，创建 `01-split-overview.md` 索引文件。索引文件列出所有子需求的编号、名称和文件链接，按前端、后端、通用三个分类组织。

5. **展示并确认**：向用户展示拆分总览内容，包括需求总数、各分类数量、关键不确定项。询问用户确认拆分结果是否合理，重点确认以下问题：
   - 拆分粒度是否合适
   - 是否有遗漏的页面或模块
   - 前后端边界是否准确
   - 不确定项是否需要进一步澄清

## 质量约束

以下规则为硬性要求，任何情况下不得违反。

### 来源标注

每个子需求必须标注来源信息，明确指向 PRD 原文的哪一节或哪一页。来源标注格式为"PRD 第 X 节 / 第 X-Y 页"。如果一个子需求的内容来自多个章节，逐一列出所有来源。来源标注信息写在需求文件的基本信息章节中，确保每个需求都可追溯到原始 PRD 的对应位置。

### 不确定项标记

对 PRD 中表述模糊、存在歧义或信息不完整的内容，统一标记为"不确定项"，列入需求文件的"不确定项"章节。禁止对不确定内容进行猜测或编造。不确定项使用待办列表格式（`- [ ] 待澄清问题`），便于后续逐项确认。常见的不确定情况包括：业务规则未明确、交互流程存在分支但未说明条件、数据字段类型或格式未定义等。

### 过滤内容保留

所有从原始 PRD 中过滤掉的内容必须完整保留在 `00-prd-raw.md` 的附录区，不得丢弃。附录区按原文章节顺序组织，每段过滤内容标注对应的原始章节标题。保留过滤内容的目的是确保信息可追溯，避免因误判而丢失重要需求信息。

### 去噪保留率

去噪处理后，保留的核心信息应覆盖原始 PRD 中 90% 以上的需求相关内容。执行去噪时采用保守策略：对于边界模糊的内容（不确定是否为需求相关信息），优先保留而非过滤。宁可多保留一些干扰信息，也不遗漏核心需求。去噪完成后进行自检，对比原始 PRD 和去噪结果，确认核心功能描述、业务规则、交互流程、数据模型等关键内容未被误删。

### 术语一致性

全文使用统一的术语体系。同一概念在整个需求拆分结果中使用相同的中文名称或英文缩写，避免同一概念在不同文件中出现不同叫法。首次出现非中文术语时，在括号中附上英文原文，例如"单点登录（Single Sign-On, SSO）"。后续出现时直接使用统一后的名称。术语的命名以 PRD 原文中的用法为优先标准，仅当 PRD 中同一概念使用了不同名称时，选择最准确的一种并在全文统一。

## 输出规范

### 输出目录

所有输出文件存放在以下路径：

```
requirements/<project-name>/
├── 00-prd-raw.md              # 去噪后 PRD + 过滤附录
├── 01-split-overview.md       # 需求拆分总览索引
├── fe-<page-name>.md          # 前端需求文件（按页面拆分）
├── be-<module-name>.md        # 后端需求文件（按模块拆分）
└── cm-<name>.md               # 通用需求文件
```

### 文件格式

- 格式：Markdown，UTF-8 编码
- 模板：子需求文件遵循 `references/requirement-template.md` 模板的章节结构
- 索引文件遵循 `references/split-overview-template.md` 模板的章节结构
- 文件命名统一使用 kebab-case 格式

### 向用户展示

完成所有文件生成后，向用户展示以下内容：

1. 输出目录的完整路径。
2. 拆分总览（即 `01-split-overview.md` 的完整内容），包括前端需求数量、后端需求数量、通用需求数量。
3. 所有不确定项的汇总列表，提示用户需要进一步澄清的问题。
4. 询问用户对拆分结果的确认意见，等待用户反馈后再结束流程。

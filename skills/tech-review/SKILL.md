# tech-review 技能说明

## 概述

本技能的目的是审阅所有已生成的调研文档，检查准确性、完整性、一致性，确保最终输出质量。审阅范围涵盖技术发现报告（01-discovery.md）、技术分析报告（02-analysis.md）和学习教程（03-tutorial.md），按七项审阅清单逐项检查，对发现的问题分级（P0/P1/P2），修复所有 P0 和 P1 问题，最终生成审阅报告。

## 前置条件

开始审阅之前，必须读取 `research/<tech-name>/` 目录下所有已生成的文档：

1. `research/<tech-name>/01-discovery.md`——技术发现报告
2. `research/<tech-name>/02-analysis.md`——技术分析报告
3. `research/<tech-name>/03-tutorial.md`——学习教程

如果上述文件缺少任何一个，提示用户先运行对应的前置技能完成报告生成，再执行审阅。若用户明确只审阅部分文档，以用户指定范围为准，但需在审阅报告中注明审阅范围。

## 执行流程

### 第一步：读取全部已生成文档

使用 Read 工具读取 `research/<tech-name>/` 下所有文档（01 至 03），同时读取 `references/review-checklist.md` 审阅清单文件作为检查依据。通读全部内容，建立对调研成果的全局理解。

### 第二步：按审阅清单逐项检查

参照 `references/review-checklist.md` 中的 7 项清单逐项检查：

- **事实准确性**：验证技术声明是否有来源支撑，版本号与官方最新版本是否一致，性能数据是否有出处。
- **代码可运行性**：检查代码示例语法是否正确，是否包含完整导入和依赖，预期输出是否与代码逻辑一致。不确定的代码通过 WebSearch 查找官方文档比对。
- **内容完整性**：检查 discovery 是否覆盖所有基本信息项，analysis 是否覆盖所有核心概念，tutorial 是否覆盖从入门到高级的关键知识点，横向对比是否包含至少 3 个竞品。
- **逻辑递进**：检查 tutorial 章节是否由浅入深，每个知识点是否建立在前一个之上，实战项目是否综合运用了教程中的知识。
- **术语一致性**：检查同一概念是否全文使用统一术语，中英文术语对应是否一致，代码命名与文字描述是否对应。
- **时效性**：检查信息是否基于最新版本，获取日期是否标注，过时信息是否已标记。
- **来源可溯**：检查关键信息是否标注来源链接，是否至少 3 个独立信息来源，来源链接是否可访问。

### 第三步：对发现的问题进行分级

将所有发现的问题按以下标准分级：

- **P0（必须修复）**：事实错误、代码无法运行、严重遗漏
- **P1（应当修复）**：重要信息不完整、代码示例不清晰、对比分析缺失维度
- **P2（建议改进）**：表述优化、排版改进、补充资源

为每个问题记录：问题级别、所在文件、具体位置、问题描述、建议修复方式。

### 第四步：修复 P0 和 P1 问题

针对 P0 和 P1 问题逐一修复。使用 WebSearch 搜索正确信息来源，确认修复方向。使用 WebFetch 抓取验证页面内容，确认修复信息准确。禁止凭记忆修复，每条修正必须有可追溯来源。直接修改对应的源文件（01 至 03），在修改处添加审阅标注 `<!-- reviewed: 修正原因 -->` 以便追溯。

### 第五步：生成审阅报告

修复完成后生成审阅报告，包含以下内容：

- **审阅清单结果**：7 项清单每项的检查结论（通过/不通过）及具体发现
- **问题列表**：表格列出所有问题，包含列：问题编号、级别、所在文件、问题描述、状态
- **修正说明**：每个已修复问题的修正内容、修正依据来源 URL、修正前后差异
- **质量评分**：基于问题数量的评级——A 级（无 P0/P1，P2 不超过 2 个）/ B 级（无 P0/P1）/ C 级（有 P1 已修复，无 P0）/ D 级（有 P0 或 P1 未修复）

### 第六步：保存审阅报告

将审阅报告保存到 `research/<tech-name>/04-review.md`。保存完成后向用户报告审阅结果，包括质量评分、主要发现和已执行的修正操作。

## 质量约束

执行审阅时必须严格遵守以下五条硬性规则：

1. **所有 P0 问题必须修复后才能输出最终版**。发现 P0 问题后必须在输出最终审阅报告之前完成修复。若因客观原因无法修复，须在审阅报告中醒目标注，状态标记为"无法修复-需人工确认"。

2. **所有 P1 问题必须修复后才能输出最终版**。P1 问题与 P0 同等对待，必须在最终输出前完成修复，不得以"建议改进"为由跳过。

3. **修正时必须通过 WebSearch 或 WebFetch 验证，不得凭记忆**。技术信息更新频繁，历史知识可能已过时。每次修正都须在审阅报告中标注验证来源的 URL。

4. **质量评分基于 P0/P1/P2 问题数量严格执行**：A（无 P0/P1，不超过 2 个 P2）/ B（无 P0/P1）/ C（有 P1 已修复，无 P0）/ D（有 P0 或 P1 未修复）。D 级报告须在开头醒目标注"本报告质量未达标"。

5. **审阅报告中必须包含每项清单的检查结论和具体发现**。不允许出现"已检查"但无具体发现的空洞结论，即使全部通过也须说明检查了哪些内容和采用的验证方式。

## 输出规范

- **审阅报告路径**：`research/<tech-name>/04-review.md`
- **报告格式**：Markdown，UTF-8 编码
- **问题列表**：使用标准 Markdown 表格格式
- **修正标注**：在源文件中使用 HTML 注释标注修正位置，格式为 `<!-- reviewed: 修正原因 -->`
- **向用户展示**：保存完成后输出审阅报告路径、质量评分、问题统计摘要和主要修正内容

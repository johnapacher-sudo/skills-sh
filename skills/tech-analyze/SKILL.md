# tech-analyze 技能

## 概述

本技能用于对指定技术进行深度分析。基于前置技能 `tech-discover` 产出的发现报告，进一步挖掘技术原理、架构设计、核心概念，并进行同类技术横向对比，最终输出一份完整的技术深度分析报告。本技能是技术调研流水线的第二阶段，承接发现阶段的初步信息，向下游提供可供决策的深度分析数据。

## 前置条件

在执行本技能之前，必须确认以下条件已满足：

1. 已通过 `tech-discover` 技能完成技术发现，且发现报告已保存至 `<tech-name>-research/01-discovery.md`。
2. 发现报告内容完整，至少包含技术简介、核心特性、官方资源链接和初步评估等关键信息。
3. 如果发现报告不存在或内容不完整，提示用户先执行 `tech-discover` 技能。

执行开始时，使用 `Read` 工具读取 `<tech-name>-research/01-discovery.md`，提取以下关键信息：技术名称与定位、核心特性列表、官方资源链接、发现报告中标注的需要深入了解的知识盲点、初步评估中提到的疑问点。将这些信息作为后续分析的基础输入。

## 执行流程

### 第一步：读取并解析发现报告

读取 `<tech-name>-research/01-discovery.md`，识别以下要点：

- 技术的官方名称、定位和核心定位描述
- 列出的核心特性，标记出需要深入分析的特性
- 官方文档链接、GitHub 仓库地址、关键博客文章链接
- 初步评估中提到的优势、劣势、疑问点
- 报告中明确标注的"待深入调研"或"需进一步验证"的内容

将识别出的要点整理为分析任务清单。

### 第二步：搜索补充信息

根据分析任务清单，使用 `WebSearch` 工具分批搜索以下类型的信息：

**架构与原理类：**
- 搜索 `<tech-name> architecture design`，获取架构设计文档和解读文章
- 搜索 `<tech-name> internals deep dive`，获取内部实现机制的深度分析
- 搜索 `<tech-name> 源码分析` 或 `<tech-name> 源码解读`，获取中文源码分析文章

**性能与评测类：**
- 搜索 `<tech-name> benchmark performance`，获取性能基准测试数据
- 搜索 `<tech-name> performance comparison`，获取与同类技术的性能对比
- 搜索 `<tech-name> production experience`，获取生产环境使用经验

**对比与生态类：**
- 搜索 `<tech-name> vs <alternative>`，获取同类技术对比文章
- 搜索 `<tech-name> ecosystem plugins`，了解生态丰富程度
- 搜索 `<tech-name> case study enterprise`，获取企业采用案例

**深度文章类：**
- 搜索技术博客上的深度文章（如 Medium、Dev.to、InfoQ、掘金等平台）
- 搜索技术会议演讲资料（如 QCon、GopherCon 等相关技术大会）
- 搜索 GitHub 上的 awesome 列表和官方 example 仓库

每次搜索后，使用 `mcp__web_reader__webReader` 抓取排名前 3 的相关页面内容，提取关键信息并记录来源 URL。确保每条关键结论都有可追溯的信息来源。

### 第三步：分析技术原理和架构

基于收集到的信息，进行以下分析：

**设计哲学：**
- 归纳该技术的核心理念和设计原则
- 分析其设计取舍（trade-off），如性能 vs 易用性、灵活 vs 约定
- 追溯其设计灵感来源（是否受其他技术启发）

**核心机制：**
- 深入分析核心算法或工作机制，用通俗语言解释其运作方式
- 绘制或描述数据流/执行流程，从输入到输出的完整链路
- 标注关键数据结构和核心抽象

**架构分层：**
- 描述整体架构分层或模块划分
- 列出核心模块及其职责
- 分析模块间的依赖关系和交互方式
- 说明扩展机制（插件系统、钩子、中间件等）

### 第四步：详解核心概念并编写代码示例

从发现报告和补充资料中提取关键概念，对每个概念进行详尽解释：

- **定义：** 用简洁语言给出准确定义
- **作用：** 说明该概念在整体架构中的角色和意义
- **使用场景：** 描述典型的应用场景和使用模式
- **代码示例：** 编写或引用可运行的代码片段

代码示例要求：
- 优先从官方文档或权威教程中引用，并标注来源
- 如需自行编写，确保基于最新稳定版本的 API
- 每个代码示例必须包含注释说明关键步骤
- 标注示例所基于的技术版本号

### 第五步：同类技术横向对比

识别该技术所属领域的 2-3 个主要竞品，构建对比表格。对比维度必须覆盖以下方面：

- 核心理念与设计哲学
- 性能表现（附具体数据或基准测试来源）
- 易用性与开发体验
- 生态丰富度（插件数量、社区活跃度）
- 社区规模（GitHub Stars、贡献者数量、Stack Overflow 问题数）
- 学习曲线
- 生产就绪度（稳定性、监控支持、故障恢复）
- 适用场景

对比数据必须来自可验证的公开信息源，标注数据获取时间。对于无法获取确切数据的维度，标注"待验证"并说明估算依据。

### 第六步：评估适用场景、生产就绪度与学习曲线

**适用场景评估：**
- 列出 3-5 个最佳适用场景，逐一说明原因
- 列出 2-3 个不适用场景，说明替代方案
- 给出典型项目规模和团队规模下的推荐程度

**生产就绪度评估：**
- 评估稳定性（是否有已知重大 Bug、发布频率、版本管理策略）
- 评估性能表现（是否有生产环境性能数据、是否存在性能瓶颈）
- 评估监控和可观测性（是否支持主流监控工具、日志和指标输出能力）
- 评估故障恢复机制（容错设计、降级策略、备份方案）
- 评估安全合规（已知安全漏洞历史、安全更新响应速度）

**学习曲线评估：**
- 列出前置知识要求（需要掌握哪些基础技术）
- 估算入门时间（能完成基本功能开发所需时间）
- 估算精通时间（能独立处理复杂场景和性能优化所需时间）

### 第七步：格式化输出并保存

使用 `Read` 工具读取 `references/output-template.md` 模板，按照模板结构将所有分析结果填充为完整报告。确保报告各章节内容完整、逻辑连贯、数据准确。

使用 `Write` 工具将最终报告保存至 `<tech-name>-research/02-analysis.md`。

## 质量约束

在执行全部分析流程时，必须严格遵守以下五条硬性规则：

1. **代码示例必须标注来源。** 所有代码示例必须基于官方文档或实际测试。基于官方文档的示例标注"基于官方文档 vX.Y.Z"，基于实际测试的示例标注"基于实际测试（环境：OS/版本，运行时版本）"。不得使用未经验证的代码片段。

2. **必须标注分析所基于的具体版本号。** 报告开头的"信息来源与版本说明"章节必须注明分析所基于的技术版本号。如果涉及多个版本（如对比不同版本差异），逐一列出。所有版本号必须是已正式发布的稳定版本，不得使用 alpha、beta 或 nightly 版本作为主要分析对象。

3. **对比维度必须一致，数据可查。** 横向对比表格中的每个维度必须适用于所有参与对比的技术，不得出现某技术在某维度留空而其他技术有数据的情况。所有定量数据必须附带可验证的来源链接或标注数据获取日期。

4. **不确信的结论必须标注置信度。** 对于无法通过权威来源完全验证的结论，在结论后标注置信度级别：`[置信度：高]` 表示有多个独立来源佐证，`[置信度：中]` 表示有部分证据支持但缺乏交叉验证，`[置信度：低]` 表示基于推理或少量证据的判断。不得将低置信度的结论作为决策依据呈现。

5. **术语全文统一，首次出现非中文术语附英文原文。** 同一概念在全文中使用统一的译名，不得交替使用不同翻译。非中文术语首次出现时必须在括号中附上英文原文，例如："响应式编程（Reactive Programming）"。后续出现时使用统一的译名即可。建立术语对照表，确保全文一致。

## 输出规范

最终输出文件保存路径：`<tech-name>-research/02-analysis.md`

输出文件格式要求：
- 使用标准 Markdown 格式
- 文件编码为 UTF-8
- 一级标题为技术名称加"深度分析报告"后缀
- 各章节严格遵循 `references/output-template.md` 的结构顺序
- 表格使用标准 Markdown 表格语法
- 代码块标注语言类型（如 ```go、```yaml）
- 所有外部链接使用标准 Markdown 链接语法，确保 URL 可访问

输出完成后，在控制台打印报告摘要，包括：技术名称、分析基于的版本、报告保存路径、报告字数。

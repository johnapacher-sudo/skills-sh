# tech-research — 技术调研总调度

## 概述

本技能是技术调研流水线的总调度入口。当用户提出技术调研需求时，自动协调 discover → analyze → tutorial → review 四个阶段，一键产出完整、高质量、全中文的技术调研报告与实战教程。

作为总调度器，本技能负责：解析用户意图、管理执行顺序、控制输出质量、汇报执行进度、确保最终交付物符合标准。所有子技能按严格顺序执行，前序阶段的输出是后续阶段的输入，形成完整的调研流水线。

## 执行流程

### Phase 1 - 准备

**提取技术名称：** 从用户输入中识别待调研的技术。用户输入可能是以下形式之一：

- 关键词（如 "WebAssembly"、"Rust"）
- GitHub URL（如 "https://github.com/vercel/next.js"）
- 技术全称（如 "React Server Components"）
- 中英文混合表述（如 "调研一下 Bun 这个运行时"）

**标准化目录名：** 将提取到的技术名称转换为 kebab-case 格式，作为调研输出目录名。转换规则如下：

- 统一使用小写字母
- 空格和特殊字符替换为连字符 `-`
- 去除连续连字符
- 示例："React Server Components" → `react-server-components`，"Kubernetes" → `kubernetes`

**创建输出目录：** 在项目根目录下创建 `research/<tech-name>/` 目录，用于存放所有调研输出文件。若目录已存在，向用户确认是否覆盖或追加内容。

### Phase 2 - 执行流水线

按以下四个阶段严格顺序执行。每个阶段的输出文件写入 `research/<tech-name>/` 目录。

#### 第一阶段：技术发现与检索（tech-discover）

执行技术发现与检索技能，全面收集目标技术的基础信息。要求如下：

- 从至少 3 个维度搜索信息：官方文档与源码、社区评测与讨论、行业应用与案例
- 对多个来源的信息进行交叉验证，标注信息可靠性等级
- 覆盖以下方面：技术定位与核心价值、发展历史与版本演进、生态系统与社区活跃度、适用场景与局限性
- 按标准模板格式输出 `01-discovery.md`

#### 第二阶段：深度分析（tech-analyze）

基于发现报告，执行深度分析技能，深入剖析技术本质。要求如下：

- 分析核心原理与设计哲学，解释技术背后的动机与权衡
- 梳理架构设计与关键模块，绘制组件关系与数据流向
- 提取核心概念，用类比和实例帮助理解
- 横向对比同类技术（至少 2 个竞品），从性能、生态、学习曲线、社区支持等维度进行表格化对比
- 按标准模板格式输出 `02-analysis.md`

#### 第三阶段：硬核教程生成（tech-tutorial）

基于前两份报告，执行硬核教程生成技能，设计从零到一的实战教程。要求如下：

- 设计渐进式学习路径，从环境搭建到高级用法逐步递进
- 每个知识点配备完整、可运行的代码示例，禁止使用伪代码或省略号占位
- 包含一个完整的实战项目，覆盖核心功能点
- 标注常见陷阱与调试技巧
- 按标准模板格式输出 `03-tutorial.md`

#### 第四阶段：内容质量审阅（tech-review）

执行内容质量审阅技能，对已生成的三份文档进行系统性审查。要求如下：

- 按审阅清单逐项检查：事实准确性、技术完整性、代码可运行性、格式规范性、中文表达质量
- 对发现的问题按严重程度分级：P0（事实错误、代码无法运行）、P1（重要遗漏、表述不清）、P2（格式优化、建议改进）
- 输出 `04-review.md`，包含问题清单与修复建议

### Phase 3 - 收尾

**回溯修复：** 检查 review 报告中标注的 P0 和 P1 问题。若存在此类问题，回溯到对应文档进行修复，修复后重新审阅确认问题已解决。重复此过程直到所有 P0/P1 问题清零。

**输出完成汇总：** 向用户输出调研完成的汇总信息，包含以下内容：

- 技术名称
- 输出目录路径
- 生成的文件列表（含每个文件的简要描述）
- 质量评分（来自 review 阶段）
- P0/P1 问题数量（修复后应为 0）

## 质量约束

以下规则贯穿全流程，任何阶段不得违反：

- **全中文输出：** 所有文档、注释、说明均使用中文撰写。技术专有名词可保留英文，但需附带中文解释
- **禁止编造信息：** 所有陈述必须有据可查。若信息不确定或无法验证，必须明确标注"待验证"，并说明待验证的原因
- **标注来源：** 每个关键信息点（数据、结论、引用）必须标注信息来源，包括 URL、文档版本、获取时间
- **代码完整可运行：** 所有代码示例必须完整，包含必要的 import 语句、依赖声明和环境配置。禁止使用 `// ...` 或 `/* 省略 */` 等占位符省略关键代码
- **P0/P1 必须修复：** 所有 P0 级别（事实错误、代码故障）和 P1 级别（重要遗漏、关键表述不清）问题必须在交付前修复完毕
- **进度实时通知：** 每个阶段完成后，立即向用户简要汇报当前进度，不得积压到最终才通知

## 进度汇报

在执行流水线过程中，按以下格式向用户实时汇报进度。每个阶段开始时输出进行中状态，完成时输出完成状态：

```
🔍 第一阶段：技术发现与检索...
✅ 01-discovery.md 已生成

🔬 第二阶段：深度分析...
✅ 02-analysis.md 已生成

📚 第三阶段：硬核教程生成...
✅ 03-tutorial.md 已生成

🔎 第四阶段：质量审阅...
✅ 04-review.md 已生成

🎯 调研完成！
```

若某个阶段耗时较长，每隔 30 秒向用户输出一次当前工作状态摘要，避免用户长时间无反馈。

若某个阶段执行失败，立即向用户报告失败原因，并提供以下选项：
- 重试当前阶段
- 跳过当前阶段继续执行
- 终止整个流水线

## 注意事项

- 若用户输入的技术名称模糊或存在歧义，先向用户确认具体调研范围再开始执行
- 若目标技术过于宽泛（如"前端框架"），建议用户缩小范围或分多次调研
- 全流程中若遇到网络访问受限导致信息获取不完整，在文档中明确标注受限部分
- 所有文档使用 UTF-8 编码，Markdown 格式，遵循统一的排版风格
